Lesson 14

Integration Project: Autocomplete Search Engine

This capstone lesson connects strings, maps, trees, DFS, heaps, sorting, testing, and complexity analysis in one Java project. The goal is not to build a production search engine. The goal is to learn how engineers choose data structures from the operations they need to support.

Example first: why autocomplete is a DSA problem

Imagine a command-line program with a dictionary of searchable phrases. When the user types "app", the program should return suggestions such as "apple", "application", and "apply". If some words have been searched more often, those should appear first.

The simplest approach stores all words in an array and scans every word for each query. 

static List<String> naiveSuggestions(List<String> words, String prefix) {
    List<String> result = new ArrayList<>();
    for (String word : words) {
        if (word.startsWith(prefix)) {
            result.add(word);
        }
    }
    return result;
}

This is easy to write, but every query costs O(n * p) in the worst case, where n is the number of words and p is the prefix length checked by startsWith. If the dictionary grows, scanning everything becomes wasteful.

The project question is: can we make prefix lookup depend mostly on the prefix length instead of the number of stored words?

Operations before data structures

A common beginner mistake is to pick a data structure because it sounds advanced. A better engineering habit is to list operations first, then choose structures that make those operations cheap.

OperationQuestionUseful structure
Add a termCan we insert a word character by character?Trie
Exact lookupDoes this complete word exist?Trie or HashSet
Prefix lookupWhich words start with this prefix?Trie plus DFS
Frequency trackingHow often was a word selected?HashMap
Top-k rankingWhich k suggestions have highest score?PriorityQueue or sorting

Project interface

Start with an interface. This separates the public behavior from the implementation. A future implementation could use a sorted array, a trie, or a database, but users of the class should not need to know that. 

public interface Autocomplete {
    void add(String term);
    boolean contains(String term);
    void recordSelection(String term);
    List<String> suggestions(String prefix, int limit);
}

The method recordSelection lets the engine learn from usage. If "application" is selected often, it should rank above less common terms with the same prefix.

Trie node design

A trie is a tree where each edge represents a character. Words with the same prefix share the same path. Each node needs child links, a marker for complete words, and enough data to recover or rank the term. 

class TrieNode {
    Map<Character, TrieNode> children = new HashMap<>();
    boolean isTerm;
    String term;
}

Storing term only at terminal nodes makes collection easier: when DFS reaches a node where isTerm is true, it can add node.term directly.

Basic implementation

The first version supports insertion, exact lookup, and finding the node that represents a prefix. 

public class TrieAutocomplete implements Autocomplete {
    private final TrieNode root = new TrieNode();
    private final Map<String, Integer> frequency = new HashMap<>();

    @Override
    public void add(String term) {
        String cleaned = normalize(term);
        if (cleaned.isEmpty()) return;

        TrieNode cur = root;
        for (char ch : cleaned.toCharArray()) {
            cur = cur.children.computeIfAbsent(ch, key -> new TrieNode());
        }
        cur.isTerm = true;
        cur.term = cleaned;
        frequency.putIfAbsent(cleaned, 0);
    }

    @Override
    public boolean contains(String term) {
        TrieNode node = findNode(normalize(term));
        return node != null && node.isTerm;
    }

    private TrieNode findNode(String text) {
        TrieNode cur = root;
        for (char ch : text.toCharArray()) {
            cur = cur.children.get(ch);
            if (cur == null) return null;
        }
        return cur;
    }

    private String normalize(String text) {
        if (text == null) return "";
        return text.trim().toLowerCase();
    }
}

Inserting a term of length L takes O(L) expected time because each character causes one hash map lookup or insertion. Exact lookup also takes O(L) expected time.

Collecting all prefix matches with DFS

Once we find the node for a prefix, every complete term below that node is a valid suggestion. This is just DFS on a tree. 

private void collectAll(TrieNode node, List<String> out) {
    if (node.isTerm) {
        out.add(node.term);
    }

    for (TrieNode child : node.children.values()) {
        collectAll(child, out);
    }
}

This traversal costs O(m), where m is the number of trie nodes in the prefix subtree visited by the search. Returning all matches can be expensive if the prefix is short, such as "a".

Ranking suggestions by frequency

A plain trie finds matching terms, but it does not decide which suggestions are most useful. We can track how often each term is selected with a HashMap<String, Integer>. 

@Override
public void recordSelection(String term) {
    String cleaned = normalize(term);
    if (!contains(cleaned)) {
        add(cleaned);
    }
    frequency.put(cleaned, frequency.getOrDefault(cleaned, 0) + 1);
}

A simple ranked implementation can collect all matches, then sort them by frequency descending and lexicographic order ascending. 

@Override
public List<String> suggestions(String prefix, int limit) {
    if (limit <= 0) return List.of();

    TrieNode start = findNode(normalize(prefix));
    if (start == null) return List.of();

    List<String> matches = new ArrayList<>();
    collectAll(start, matches);

    matches.sort((a, b) -> {
        int byFrequency = Integer.compare(frequency.getOrDefault(b, 0), frequency.getOrDefault(a, 0));
        if (byFrequency != 0) return byFrequency;
        return a.compareTo(b);
    });

    if (matches.size() <= limit) return matches;
    return new ArrayList<>(matches.subList(0, limit));
}

If there are s matching terms, sorting costs O(s log s). This is perfectly fine for a first project version because it is simple and correct.

Top-k with a priority queue

If the prefix has many matches but the user only wants k suggestions, a heap can avoid sorting every match. Keep a min-heap of size at most k. The weakest suggestion stays at the top and gets removed when a stronger one appears. 

private List<String> topK(List<String> matches, int k) {
    PriorityQueue<String> heap = new PriorityQueue<>((a, b) -> {
        int byFrequency = Integer.compare(frequency.getOrDefault(a, 0), frequency.getOrDefault(b, 0));
        if (byFrequency != 0) return byFrequency;
        return b.compareTo(a);
    });

    for (String term : matches) {
        heap.add(term);
        if (heap.size() > k) {
            heap.remove();
        }
    }

    List<String> result = new ArrayList<>();
    while (!heap.isEmpty()) {
        result.add(heap.remove());
    }
    Collections.reverse(result);
    return result;
}

This costs O(s log k) after collecting matches, where s is the number of matching terms. Sorting costs O(s log s). The heap version is better when k is much smaller than s.

Putting the pieces together

A complete version can use sorting first because it is easier to reason about. Then the project can add the heap version as an optimization. 

public List<String> suggestionsWithHeap(String prefix, int limit) {
    if (limit <= 0) return List.of();

    TrieNode start = findNode(normalize(prefix));
    if (start == null) return List.of();

    List<String> matches = new ArrayList<>();
    collectAll(start, matches);
    return topK(matches, limit);
}

This method combines several previous lessons: trie nodes are tree nodes, collectAll is DFS, frequency is a hash map, and topK uses a heap.

Command-line demo

A small CLI makes the project easier to test manually. It does not need to be fancy. The important part is that the program exercises the public interface instead of reaching into private fields. 

public class Main {
    public static void main(String[] args) {
        Autocomplete engine = new TrieAutocomplete();
        engine.add("apple");
        engine.add("application");
        engine.add("apply");
        engine.add("banana");

        engine.recordSelection("application");
        engine.recordSelection("application");
        engine.recordSelection("apple");

        System.out.println(engine.suggestions("app", 3));
    }
}
// Expected output:
// [application, apple, apply]

Testing strategy

A capstone project needs tests because multiple data structures interact. Test behavior through the interface. Do not test private implementation details.

Test caseWhy it matters
Empty prefixDecide whether it returns top terms or all terms up to the limit.
No matching prefixShould return an empty list, not crash.
Duplicate addShould not create duplicate suggestions.
Case and spaces" Apple " and "apple" should behave consistently if normalized.
Frequency tieShould use lexicographic order or another documented tie-breaker.
Limit smaller than matchesShould return exactly the best k suggestions.
 
static void expectEquals(Object expected, Object actual) {
    if (!Objects.equals(expected, actual)) {
        throw new AssertionError("expected " + expected + " but got " + actual);
    }
}

static void testRanking() {
    Autocomplete engine = new TrieAutocomplete();
    engine.add("apple");
    engine.add("application");
    engine.add("apply");

    engine.recordSelection("apply");
    engine.recordSelection("apply");
    engine.recordSelection("apple");

    expectEquals(List.of("apply", "apple"), engine.suggestions("app", 2));
}

Complexity summary

OperationTimeSpaceNotes
add(term)O(L) expectedO(L) new nodes worst caseL is term length.
contains(term)O(L) expectedO(1)One child lookup per character.
Find prefix nodeO(P) expectedO(1)P is prefix length.
Collect matchesO(M)O(H + S)M visited nodes, H recursion depth, S results.
Rank by sortingO(S log S)O(S)Good first implementation.
Rank by heapO(S log k)O(k)Better when k is small.

Design tradeoffs

  • HashMap children vs array children: a HashMap supports any character set and saves space for sparse nodes; an array can be faster for fixed lowercase English letters.
  • Store term at terminal node: uses more memory but simplifies DFS output.
  • Sort all matches vs heap top-k: sorting is simpler; heap is better when only a small number of suggestions is needed.
  • Normalize input or preserve original text: normalization makes lookup consistent; preserving original text may be better for display.
  • Recursive DFS vs iterative DFS: recursion is concise; iterative traversal avoids stack overflow for very deep tries.

  • Use a trie for insertion, exact lookup, and prefix search.

Extension options
