Lesson 10

Sorting Algorithms and Sorting as a Tool

Sorting is both a family of algorithms and a problem-solving technique. After data is sorted, binary search, two pointers, duplicate detection, interval merging, and many greedy algorithms become easier.

Example first: merging intervals

Given meeting intervals, merge all overlapping intervals. The key first step is sorting by start time. 

static int[][] mergeIntervals(int[][] intervals) {
    Arrays.sort(intervals, Comparator.comparingInt(a -> a[0]));
    ArrayList<int[]> merged = new ArrayList<>();

    for (int[] cur : intervals) {
        if (merged.isEmpty() || merged.get(merged.size() - 1)[1] < cur[0]) {
            merged.add(cur);
        } else {
            int[] last = merged.get(merged.size() - 1);
            last[1] = Math.max(last[1], cur[1]);
        }
    }
    return merged.toArray(new int[merged.size()][]);
}

Sorting creates order. Once intervals are sorted, we only compare the current interval with the last merged interval. The sorting cost dominates: O(n log n).

Sorting vocabulary

TermMeaning
In-placeUses O(1) or small auxiliary space, not counting input
StableEqual keys keep their original relative order
Comparison sortOnly learns order by comparing pairs
AdaptiveRuns faster on already-sorted or nearly-sorted data
OnlineCan process data as it arrives

Stability matters when records have multiple fields. If students are sorted by last name and then stably sorted by graduation year, students with the same year remain ordered by last name.

Comparison sorting lower bound

Any comparison-based sorting algorithm has a worst-case lower bound of Ω(n log n). The intuition comes from decision trees. There are n! possible input orderings, and comparisons split the possibilities into branches. A binary decision tree needs height at least log2(n!). 

log(n!) = log(1) + log(2) + ... + log(n)
        = Θ(n log n)

Therefore, algorithms like merge sort and heap sort are asymptotically optimal among comparison sorts. Non-comparison sorts can beat n log n only when they use extra assumptions about the keys.

Bubble sort

Bubble sort repeatedly swaps adjacent out-of-order elements. Large elements "bubble" to the end. It is mostly useful for learning invariants, not for production code. 

static void bubbleSort(int[] a) {
    for (int end = a.length - 1; end > 0; end--) {
        boolean swapped = false;
        for (int i = 0; i < end; i++) {
            if (a[i] > a[i + 1]) {
                swap(a, i, i + 1);
                swapped = true;
            }
        }
        if (!swapped) return;
    }
}

With the swapped optimization, the best case is O(n) for an already sorted array. Worst case remains O(n^2).

Selection sort

Selection sort repeatedly selects the smallest remaining item and places it at the next position. 

static void selectionSort(int[] a) {
    for (int i = 0; i < a.length; i++) {
        int min = i;
        for (int j = i + 1; j < a.length; j++) {
            if (a[j] < a[min]) min = j;
        }
        swap(a, i, min);
    }
}

Selection sort always does about n^2 / 2 comparisons, even if the array is already sorted. It performs at most n swaps, which can matter when writes are expensive.

Insertion sort

Insertion sort keeps a sorted prefix and inserts the next element into the correct place. 

static void insertionSort(int[] a) {
    for (int i = 1; i < a.length; i++) {
        int x = a[i];
        int j = i - 1;
        while (j >= 0 && a[j] > x) {
            a[j + 1] = a[j];
            j--;
        }
        a[j + 1] = x;
    }
}

Insertion sort is excellent for small arrays and nearly sorted arrays. Many real library sorts use insertion sort as a small-subarray optimization.

Merge two sorted arrays

Merge is the core operation behind merge sort. 

static int[] merge(int[] a, int[] b) {
    int[] out = new int[a.length + b.length];
    int i = 0;
    int j = 0;
    int k = 0;

    while (i < a.length && j < b.length) {
        if (a[i] <= b[j]) out[k++] = a[i++];
        else out[k++] = b[j++];
    }
    while (i < a.length) out[k++] = a[i++];
    while (j < b.length) out[k++] = b[j++];
    return out;
}

The merge step is linear because each pointer only moves forward.

Merge sort

Merge sort splits the array, sorts both halves, then merges. It is stable and guarantees O(n log n) time. 

static void mergeSort(int[] a) {
    int[] aux = new int[a.length];
    mergeSort(a, aux, 0, a.length);
}

static void mergeSort(int[] a, int[] aux, int lo, int hi) {
    if (hi - lo <= 1) return;
    int mid = lo + (hi - lo) / 2;
    mergeSort(a, aux, lo, mid);
    mergeSort(a, aux, mid, hi);
    merge(a, aux, lo, mid, hi);
}

static void merge(int[] a, int[] aux, int lo, int mid, int hi) {
    int i = lo;
    int j = mid;
    int k = lo;

    while (i < mid && j < hi) {
        if (a[i] <= a[j]) aux[k++] = a[i++];
        else aux[k++] = a[j++];
    }
    while (i < mid) aux[k++] = a[i++];
    while (j < hi) aux[k++] = a[j++];

    for (int p = lo; p < hi; p++) {
        a[p] = aux[p];
    }
}

Recurrence: T(n) = 2T(n/2) + O(n). By Master Theorem or recursion tree, T(n) = O(n log n).

Quick sort

Quick sort partitions around a pivot: smaller values go left, larger values go right. Then it recursively sorts both sides. 

static void quickSort(int[] a) {
    Random rng = new Random();
    quickSort(a, 0, a.length - 1, rng);
}

static void quickSort(int[] a, int lo, int hi, Random rng) {
    if (lo >= hi) return;
    int pivotIndex = lo + rng.nextInt(hi - lo + 1);
    swap(a, pivotIndex, hi);
    int p = partition(a, lo, hi);
    quickSort(a, lo, p - 1, rng);
    quickSort(a, p + 1, hi, rng);
}
static int partition(int[] a, int lo, int hi) {
    int pivot = a[hi];
    int write = lo;
    for (int read = lo; read < hi; read++) {
        if (a[read] <= pivot) {
            swap(a, write, read);
            write++;
        }
    }
    swap(a, write, hi);
    return write;
}

Randomized quick sort has expected O(n log n) time. Worst case is O(n^2) when pivots are consistently terrible. Randomization makes that unlikely for ordinary use.

Heap sort

Heap sort builds a max-heap, then repeatedly swaps the maximum with the end of the heap. 

static void heapSort(int[] a) {
    for (int i = a.length / 2 - 1; i >= 0; i--) {
        sinkMax(a, i, a.length);
    }

    for (int end = a.length - 1; end > 0; end--) {
        swap(a, 0, end);
        sinkMax(a, 0, end);
    }
}

static void sinkMax(int[] a, int i, int size) {
    while (true) {
        int left = 2 * i + 1;
        int right = 2 * i + 2;
        int largest = i;
        if (left < size && a[left] > a[largest]) largest = left;
        if (right < size && a[right] > a[largest]) largest = right;
        if (largest == i) break;
        swap(a, i, largest);
        i = largest;
    }
}

Heap sort guarantees O(n log n) time and uses O(1) auxiliary space, but it is not stable and is often slower in practice than highly optimized library sorts.

Counting sort

Counting sort is not comparison-based. It works when keys are integers in a small range. 

static int[] countingSortNonnegative(int[] a, int maxValue) {
    int[] count = new int[maxValue + 1];
    for (int x : a) {
        count[x]++;
    }

    int[] out = new int[a.length];
    int index = 0;
    for (int value = 0; value < count.length; value++) {
        for (int c = 0; c < count[value]; c++) {
            out[index++] = value;
        }
    }
    return out;
}

Time is O(n + k), where k is the key range. Counting sort is great when k is small; it is wasteful when k is huge.

Stable counting sort

Stable counting sort uses prefix sums over counts to place equal keys in original order. It is a building block for radix sort. 

static int[] stableCountingSort(int[] a, int maxValue) {
    int[] count = new int[maxValue + 1];
    for (int x : a) count[x]++;

    for (int i = 1; i < count.length; i++) {
        count[i] += count[i - 1];
    }

    int[] out = new int[a.length];
    for (int i = a.length - 1; i >= 0; i--) {
        int x = a[i];
        count[x]--;
        out[count[x]] = x;
    }
    return out;
}

Radix sort idea

Radix sort sorts integers digit by digit using a stable subroutine, usually counting sort. For base 10, sort by ones digit, then tens digit, then hundreds digit, and so on. 

RADIX-SORT(a):
    exp = 1
    while max(a) / exp > 0:
        stable sort by digit (a[i] / exp) % 10
        exp = exp * 10

If there are d digits and base k, runtime is O(d(n + k)). This can be linear when d and k are treated as constants.

Java sorting API

int[] nums = {5, 1, 4};
Arrays.sort(nums);

String[] names = {"Ada", "Grace", "Linus"};
Arrays.sort(names);

ArrayList<String> list = new ArrayList<>(List.of("Ada", "Grace", "Linus"));
list.sort(Comparator.comparingInt(String::length));

For primitives, Arrays.sort uses optimized algorithms suitable for primitive arrays. For object arrays and lists, Java uses stable comparison-based sorting. In normal programming, use the library. In DSA, implement sorts to understand the tradeoffs.

Sorting as preprocessing

Problem clueWhy sorting helpsFollow-up pattern
IntervalsOrder by start or endMerge or greedy scan
Pairs with target sumSorted order gives directionTwo pointers
DuplicatesEqual values become adjacentSingle scan
Closest valuesNearby values become neighborsSliding or adjacent comparison
Rank or percentileSorted position has meaningIndex lookup or binary search

Sorting rule: ask whether changing order is allowed. If original order matters, store original indices or use a different approach.

Common mistakes

  • Assuming every O(n log n) sort is stable.
  • Forgetting quick sort's O(n^2) worst case.
  • Using counting sort when the key range is too large.
  • Sorting and losing original indices needed for the answer.
  • Writing a comparator with subtraction and causing overflow.
  • Confusing in-place with stable; they are different properties.

Analysis routine

ANALYZE-SORTING(problem):
    1. Are we sorting numbers, objects, or keys with a small range?
    2. Is stability required?
    3. Is in-place sorting required?
    4. What are best, average, and worst-case times?
    5. What auxiliary space is used?
    6. Does sorting destroy information such as original order?
    7. What pattern becomes easier after sorting?
