Lesson 13

Dynamic Programming: States, Transitions, and Tables

Dynamic programming is a way to avoid solving the same subproblem over and over. The core skill is not memorizing formulas. The core skill is learning how to define a state, connect states with transitions, and compute them in a valid order.

Example first: why naive recursion becomes slow

Suppose a staircase has n steps. You can climb either 1 or 2 steps at a time. How many different ways are there to reach the top?

The natural recursive idea is simple: to reach step n, the last move came from n - 1 or n - 2. 

static int climbStairsSlow(int n) {
    if (n == 0) return 1;
    if (n < 0) return 0;
    return climbStairsSlow(n - 1) + climbStairsSlow(n - 2);
}

This code is correct, but it is slow. For example, climbStairsSlow(6) calls climbStairsSlow(4) more than once. It also calls climbStairsSlow(3), climbStairsSlow(2), and climbStairsSlow(1) many times. The recursion tree has repeated subtrees.

DP starts when two things are both true: the problem can be split into smaller subproblems, and many of those subproblems repeat.

Memoization: recursion plus a cache

Memoization keeps the recursive meaning but stores each answer after computing it once. The first call to a state does the work; later calls return from the cache. 

static int climbStairsMemo(int n) {
    int[] memo = new int[n + 1];
    Arrays.fill(memo, -1);
    return climb(n, memo);
}

static int climb(int stepsLeft, int[] memo) {
    if (stepsLeft == 0) return 1;
    if (stepsLeft < 0) return 0;
    if (memo[stepsLeft] != -1) return memo[stepsLeft];

    memo[stepsLeft] = climb(stepsLeft - 1, memo) + climb(stepsLeft - 2, memo);
    return memo[stepsLeft];
}

There are only n + 1 possible values of stepsLeft. Each one is computed once, so the time complexity becomes O(n). The memo array and recursion stack use O(n) space.

Tabulation: build the table bottom-up

Tabulation removes recursion. Instead of asking the original problem to call smaller problems, we fill smaller answers first and use them to build larger answers. 

static int climbStairsTable(int n) {
    int[] dp = new int[n + 1];
    dp[0] = 1;

    for (int i = 1; i <= n; i++) {
        dp[i] += dp[i - 1];
        if (i >= 2) dp[i] += dp[i - 2];
    }
    return dp[n];
}

Here, dp[i] means "the number of ways to reach step i." The transition says: every way to reach i - 1 can take one more step, and every way to reach i - 2 can take a two-step jump.

The four questions of DP

Before writing Java, answer these four questions in complete sentences. If any answer is vague, the code will probably be vague too.

QuestionWhat it meansClimbing stairs answer
StateWhat does one table entry represent?dp[i] is ways to reach step i.
TransitionHow do smaller states create this state?dp[i] = dp[i - 1] + dp[i - 2].
Base caseWhat answers are known immediately?dp[0] = 1.
OrderWhich states must be computed first?Compute from small i to large i.

Memoization vs tabulation

StyleBest whenRisk
MemoizationThe recursive structure is obvious, or not all states are reachable.Deep recursion can overflow the stack.
TabulationThe evaluation order is simple and all states are likely needed.It is easy to fill the table in the wrong direction.

Both styles have the same DP idea. Memoization asks, "What is the answer to this state?" Tabulation asks, "Which states can I already build?"

Pattern 1: 1D choice DP

Many DP problems process an array from left to right. At each index, we choose whether to take something, skip something, start something, or extend something.

Example: a row of houses contains money. Adjacent houses cannot both be chosen. What is the maximum money that can be collected? 

static int rob(int[] money) {
    if (money.length == 0) return 0;

    int[] dp = new int[money.length + 1];
    dp[0] = 0;
    dp[1] = money[0];

    for (int i = 2; i <= money.length; i++) {
        int skip = dp[i - 1];
        int take = dp[i - 2] + money[i - 1];
        dp[i] = Math.max(skip, take);
    }
    return dp[money.length];
}

State: dp[i] is the best answer using the first i houses. Transition: either skip house i - 1, or take it and combine it with the best answer before its neighbor.

The table version uses O(n) space, but the transition only needs the previous two values. We can compress it to O(1) extra space. 

static int robSpaceOptimized(int[] money) {
    int twoBack = 0;
    int oneBack = 0;

    for (int amount : money) {
        int cur = Math.max(oneBack, twoBack + amount);
        twoBack = oneBack;
        oneBack = cur;
    }
    return oneBack;
}

Pattern 2: grid DP

Grid DP usually means the state is a cell, such as dp[r][c]. The transition depends on the legal directions of movement.

Example: starting at the top-left corner, moving only right or down, how many paths reach the bottom-right corner? 

static int uniquePaths(int rows, int cols) {
    int[][] dp = new int[rows][cols];

    for (int r = 0; r < rows; r++) dp[r][0] = 1;
    for (int c = 0; c < cols; c++) dp[0][c] = 1;

    for (int r = 1; r < rows; r++) {
        for (int c = 1; c < cols; c++) {
            dp[r][c] = dp[r - 1][c] + dp[r][c - 1];
        }
    }
    return dp[rows - 1][cols - 1];
}

State: dp[r][c] is the number of ways to reach cell (r, c). Transition: the last move came from above or from the left.

Grid DP with costs

Instead of counting paths, sometimes we want the cheapest path. The state shape is the same, but the transition changes from addition to minimum. 

static int minPathSum(int[][] cost) {
    int rows = cost.length;
    int cols = cost[0].length;
    int[][] dp = new int[rows][cols];

    dp[0][0] = cost[0][0];
    for (int r = 1; r < rows; r++) {
        dp[r][0] = dp[r - 1][0] + cost[r][0];
    }
    for (int c = 1; c < cols; c++) {
        dp[0][c] = dp[0][c - 1] + cost[0][c];
    }

    for (int r = 1; r < rows; r++) {
        for (int c = 1; c < cols; c++) {
            dp[r][c] = cost[r][c] + Math.min(dp[r - 1][c], dp[r][c - 1]);
        }
    }
    return dp[rows - 1][cols - 1];
}

A useful observation: graph shortest path and DP can look similar. The difference is that DP normally relies on an acyclic dependency order. In this grid, every state depends only on cells above or left, so the row-major order is valid.

Pattern 3: subsequence DP

Subsequence problems often compare prefixes of strings or arrays. A subsequence keeps relative order but does not need to be contiguous.

Example: what is the length of the longest common subsequence between two strings? 

static int longestCommonSubsequence(String a, String b) {
    int[][] dp = new int[a.length() + 1][b.length() + 1];

    for (int i = 1; i <= a.length(); i++) {
        for (int j = 1; j <= b.length(); j++) {
            if (a.charAt(i - 1) == b.charAt(j - 1)) {
                dp[i][j] = 1 + dp[i - 1][j - 1];
            } else {
                dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
            }
        }
    }
    return dp[a.length()][b.length()];
}

State: dp[i][j] is the LCS length using the first i characters of a and the first j characters of b. The extra row and column represent empty prefixes, which makes the base cases clean. 

// Example:
// a = "abcde"
// b = "ace"
// LCS = "ace", length 3

Pattern 4: knapsack DP

Knapsack problems are about limited resources. Each item has a cost, and we want the best value without exceeding capacity.

In 0/1 knapsack, each item can be chosen at most once. 

static int knapsack(int[] weight, int[] value, int capacity) {
    int n = weight.length;
    int[][] dp = new int[n + 1][capacity + 1];

    for (int i = 1; i <= n; i++) {
        for (int cap = 0; cap <= capacity; cap++) {
            dp[i][cap] = dp[i - 1][cap];

            if (weight[i - 1] <= cap) {
                int take = dp[i - 1][cap - weight[i - 1]] + value[i - 1];
                dp[i][cap] = Math.max(dp[i][cap], take);
            }
        }
    }
    return dp[n][capacity];
}

State: dp[i][cap] is the best value using the first i items with capacity cap. Transition: skip item i - 1, or take it if there is enough capacity.

Time complexity is O(n * capacity). This is called pseudo-polynomial time because it depends on the numeric capacity, not only the number of input items.

Space optimization and loop direction

0/1 knapsack can be compressed to a 1D array, but the loop direction matters. We iterate capacity backward so the same item is not reused multiple times in one iteration. 

static int knapsackOneArray(int[] weight, int[] value, int capacity) {
    int[] dp = new int[capacity + 1];

    for (int i = 0; i < weight.length; i++) {
        for (int cap = capacity; cap >= weight[i]; cap--) {
            dp[cap] = Math.max(dp[cap], dp[cap - weight[i]] + value[i]);
        }
    }
    return dp[capacity];
}

If the inner loop goes forward, dp[cap - weight[i]] may already include item i, which would turn the solution into an unbounded knapsack variant by accident.

Top-down DP with non-array states

Not every DP state is a clean integer index. Sometimes a state is a pair, a string, a bitmask, or a custom object. In those cases, a HashMap memo can be easier than an array. 

static int minCoins(int[] coins, int amount) {
    Map<Integer, Integer> memo = new HashMap<>();
    int ans = minCoinsFrom(coins, amount, memo);
    return ans >= 1_000_000 ? -1 : ans;
}

static int minCoinsFrom(int[] coins, int remaining, Map<Integer, Integer> memo) {
    if (remaining == 0) return 0;
    if (remaining < 0) return 1_000_000;
    if (memo.containsKey(remaining)) return memo.get(remaining);

    int best = 1_000_000;
    for (int coin : coins) {
        best = Math.min(best, 1 + minCoinsFrom(coins, remaining - coin, memo));
    }

    memo.put(remaining, best);
    return best;
}

This is DP because remaining is the state, and many paths may ask for the same remaining amount. It is also a shortest-path-like problem where every coin is an edge from one amount to a smaller amount.

DP as a graph of dependencies

A good way to understand DP is to imagine each state as a vertex in a directed graph. If dp[i] depends on dp[i - 1], draw an edge from i to i - 1. A tabulation order is valid only if every dependency is computed before the state that needs it. 

dp[i] depends on dp[i - 1] and dp[i - 2]

0    1    2    3    4    5
          ^    ^    ^    ^
          |    |    |    |
        earlier states must already be known

This is why some DP problems are naturally left-to-right, some are right-to-left, some are diagonal across a matrix, and some are easier with memoized recursion.

How to recognize DP

  • The problem asks for an optimum: minimum, maximum, fewest, most, longest, shortest under special constraints.
  • The problem asks for a count of ways.
  • A recursive solution works but repeats many calls.
  • The decision at one index affects which future choices are legal.
  • The input has prefixes, capacities, grid positions, or remaining amounts.

These are signals, not guarantees. Some "shortest" problems are graph problems, some "maximum" problems are greedy, and some counting problems require combinatorics. The state definition is what decides whether DP is a good fit.

Common mistakes

  • Writing code before defining the state. A DP table without a clear meaning is just an array.
  • Off-by-one errors. Extra rows and columns are often worth it because empty prefixes become easy.
  • Wrong loop direction. Especially dangerous in compressed knapsack-style DP.
  • Returning the wrong cell. Know whether the answer is dp[n], max(dp), or another state.
  • Confusing subsequence and substring. Substrings are contiguous; subsequences are not.
  • Ignoring integer overflow. Counting problems can exceed int; use long if needed.

Complexity analysis routine

For DP, complexity is usually found by counting states and work per state. 

time = number of states * work per state
space = number of stored states + recursion stack if memoized

README requirements

  • For each problem, define the state in one complete English sentence.
  • Write the transition formula or transition rule.
  • List base cases and explain why they are correct.
  • Give time and space complexity using "number of states times work per state."
  • For knapsack, explain why the 1D loop must go backward.

A good DP submission should make the table meaning obvious before the code. The goal is not only to pass tests; the goal is to explain why each table entry is correct.
