Course logistics

Assignment System and GitHub Submission Guide

Homework in this course is not only about getting the right answer. Each submission should show three skills: implementing Java code, testing behavior, and explaining time and space complexity clearly.

Submission model

Use one GitHub repository for the course, with one folder per lesson. This keeps the full learning history in one place and makes it easier to compare early code with later code. 

dsa-java-homework/
  README.md
  lesson-01-java-warmup/
  lesson-02-time-complexity/
  lesson-03-space-amortization/

A single-lesson repository is also acceptable, but every assignment still needs source code, tests or a test harness, and written analysis.

README template

Every lesson folder should include a short README. The README is where the student explains the work as if a new reader is opening the repository for the first time. 

# Lesson 10 - Sorting

## What this assignment implements
- Merge sort
- Quick sort
- Insertion sort
- Sorting comparison demo

## How to run
```bash
javac src/*.java
java -cp src Main
```

## How to test
```bash
javac src/*.java tests/*.java
java -cp src:tests SortingTest
```

## Key ideas
- Merge sort divides the array and merges sorted halves.
- Quick sort partitions around a pivot.
- Insertion sort is useful for small or nearly sorted arrays.

## Complexity summary
See `analysis/complexity.md` for method-by-method analysis.

## Known limitations
- The quick sort implementation does not use randomized pivots yet.

Analysis template

Complexity analysis should define variables. Avoid writing only O(n). Explain what n means and why the bound is true. 

# Complexity Analysis

## Method: mergeSort(int[] arr)
n = arr.length

Time: O(n log n)
Reason: The array is split into two halves for about log n levels.
At each level, merging touches each element once, so the work per level is O(n).

Space: O(n)
Reason: The merge step uses temporary arrays whose total size is proportional to n.

## Method: binarySearch(int[] arr, int target)
n = arr.length

Time: O(log n)
Reason: Each comparison removes about half of the remaining search range.

Space: O(1)
Reason: The iterative version only stores left, right, and mid.

Testing expectations

Tests should show that the code works on normal cases, edge cases, and likely bug cases. A simple custom test harness is fine if the project does not use JUnit. 

import java.util.*;

public class SortingTest {
    static void expectArrayEquals(int[] expected, int[] actual) {
        if (!Arrays.equals(expected, actual)) {
            throw new AssertionError(
                "expected " + Arrays.toString(expected) +
                " but got " + Arrays.toString(actual)
            );
        }
    }

    public static void main(String[] args) {
        expectArrayEquals(new int[] {}, Sorting.mergeSort(new int[] {}));
        expectArrayEquals(new int[] {1}, Sorting.mergeSort(new int[] {1}));
        expectArrayEquals(new int[] {1, 2, 3}, Sorting.mergeSort(new int[] {3, 1, 2}));
        expectArrayEquals(new int[] {1, 1, 2}, Sorting.mergeSort(new int[] {1, 2, 1}));
        System.out.println("All tests passed.");
    }
}

Git workflow

Each assignment should have more than one meaningful commit when possible. The goal is to show progress, not only the final state.

Semantic commit messages

All commits in this course should follow the Conventional Commits format. This keeps the project history scannable and makes it easy to see what each change does at a glance.

The format is:

<type>: <short description>

Use one of the following types:

TypeWhen to useExample
featAdd new functionality or implement an algorithmfeat: implement merge sort
fixFix a bug or correct wrong outputfix: handle empty array in quick sort
docsAdd or update documentation, README, or analysisdocs: write sorting complexity analysis
testAdd or update test casestest: add edge case tests for merge sort
refactorRestructure code without changing behaviorrefactor: extract partition helper for quick sort
styleCode formatting, naming, whitespace onlystyle: rename variables for clarity
choreBuild config, .gitignore, project setupchore: add .gitignore for Java class files

A scope is optional but can help when working on multiple features: feat(heap): implement sift-down operation.

Example workflow

git status
git add lesson-10-sorting
git commit -m "feat: implement merge sort"
git commit -m "test: add sorting edge case tests"
git commit -m "docs: write sorting complexity analysis"
git push

Commit messages should describe the change. Avoid messages like update, fix stuff, or homework when a more specific message is possible. The type prefix already classifies the change, so focus the description on what was done.

Commit convention: every commit message must start with a semantic type prefix. Submissions with generic messages like "update" or "changes" will lose points under the GitHub hygiene category.

Rubric

CategoryWeightExcellent submission
Correctness35%Passes expected cases, handles edge cases, and produces stable output.
Code quality20%Clear names, small methods, no unnecessary cleverness, no duplicated logic where a helper would help.
Testing15%Includes normal cases, boundary cases, duplicates, and failure cases.
Complexity analysis20%Defines variables, explains time and space, and connects the explanation to the code structure.
GitHub hygiene10%Readable folder structure, useful README, meaningful commits, no generated clutter.

Code quality checklist

  • Use method names that describe behavior, such as insert, contains, or heapify.
  • Keep public methods small enough that their main idea is visible.
  • Use private helpers for repeated logic.
  • Prefer Java collection interfaces when appropriate, such as List, Map, and Set.
  • Do not print inside library methods unless printing is part of the method's purpose.
  • Validate inputs when invalid input would cause confusing behavior.
  • Keep test data small enough to inspect by hand, then add a few larger stress cases.

Final project expectation

The final project should read like a small engineering artifact. It should have a clear interface, a working implementation, tests, and written design decisions. The best submissions explain not only what works, but why the chosen data structures are appropriate.

Submission rule: code without analysis is incomplete. Analysis without runnable code is incomplete. The homework should show both programming skill and reasoning.
