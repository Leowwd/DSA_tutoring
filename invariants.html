How to Think in Invariants

An invariant is a fact that stays true while an algorithm runs. Templates are easier to remember when you know the invariant they protect.

The invariant pattern

Before the loop or recursion:
    What is true?
During each step:
    What action preserves that truth?
After the loop or recursion:
    How does that truth imply the answer?

Core DSA invariants

TopicInvariantWhy it matters
Binary searchIf the target exists, it is inside [lo, hi].Each comparison safely discards half.
Sliding windowThe window represents the current valid or nearly valid range.Move left/right without recomputing from scratch.
Monotonic stackStack values stay increasing or decreasing.Elements that can never be answers are removed.
HeapEvery parent has priority over its children.The root is always the next item to remove.
BSTEvery subtree has a valid low/high range.Search and validation use ordering safely.
Union-FindEach node points toward a representative.Find returns component identity.
DijkstraFinalized distances are already optimal.Requires nonnegative edge weights.
DPEvery filled state has the meaning stated by the state definition.Transitions can rely on previous states.

Example: binary search

static int binarySearch(int[] a, int target) {
    int lo = 0;
    int hi = a.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (a[mid] == target) return mid;
        if (a[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return -1;
}

Invariant: if target is present, it is in the current range [lo, hi]. When a[mid] < target, every index at or left of mid is too small, so moving lo to mid + 1 preserves the invariant.

Example: DP state invariant

In DP, the invariant is the state sentence. For house robber: 

dp[i] = the maximum money using the first i houses

Every transition must preserve that meaning. If the code accidentally uses dp[i] to mean "best ending exactly at house i" in one line and "best up to i" in another, the solution becomes inconsistent.
