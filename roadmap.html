DSA Roadmap: What to Use, When, and Why

This page is the missing bridge between individual lessons. If a student can read a problem and use this page to choose a structure, pattern, and complexity argument, the course is starting to work.

The one question behind DSA

Data structures and algorithms are about making the right operation cheap. Before choosing a tool, ask: 

What data must I store?
What question must I answer?
Which operation happens most often?
What constraints make a simple solution too slow?
What invariant can I maintain?

Common confusions

ConfusionHow to separate them
ADT vs implementationAn ADT says what operations exist. An implementation decides how those operations work internally.
Array vs ArrayListAn array has fixed length. An ArrayList grows by managing a backing array.
Stack vs recursionRecursion uses the call stack. An explicit stack is a data structure you control.
BFS vs DFSBFS is layer-by-layer and good for unweighted shortest path. DFS goes deep and is good for structure exploration.
HashMap vs TreeMapHashMap is expected O(1) but unordered. TreeMap is O(log n) and ordered.
Heap vs BSTA heap gives fast priority removal but not full ordered search. A BST supports ordered search and traversal.
Greedy vs DPGreedy commits to a local choice. DP stores many possible subproblem answers.
DP vs graph shortest pathDP usually has an acyclic dependency order. Graph shortest path handles more general connectivity and weights.
Subarray vs subsequenceA subarray is contiguous. A subsequence preserves order but may skip elements.

Problem-solving workflow

  1. Restate the problem. Say the input, output, and constraints in your own words.
  2. Try the brute force solution. It reveals the operation that is too expensive.
  3. Name the expensive operation. Search, insert, delete, min/max, range query, connectivity, or repeated subproblem.
  4. Choose a structure or pattern. Use the tables above, then justify the choice.
  5. Define the invariant. Examples: window is valid, heap parent is smaller, BST left is smaller, DP state means a specific subproblem.
  6. Write pseudocode. Keep it language-light before Java syntax gets in the way.
  7. Implement in Java. Use clear helper methods and test small cases.
  8. Analyze complexity. Define variables and explain where the bound comes from.

What "understanding" should feel like

A student understands a topic when they can do four things without copying a template:

  • Explain why the structure or algorithm exists.
  • Trace a small example by hand.
  • Implement the core operation in Java.
  • Analyze time and space complexity with defined variables.

If a solution feels like memorization, go back to the operation. DSA becomes clearer when every tool is tied to the operation it makes cheaper.
